# equals() and hashCode() Contract  
  
Java provides `equals()` and `hashCode()` methods through the `Object` class.  
  
These methods are important when working with hash-based collections such as:  
  
- HashMap  
- HashSet  
- Hashtable  
  
---  
  
## Why They Matter  
  
Hash-based collections use both methods to store and retrieve elements efficiently.  
  
The general process used by HashMap is:  
  
1. Compute `hashCode()` of the key  
2. Determine the bucket index  
3. Traverse the bucket  
4. Use `equals()` to identify the correct key  
  
This allows constant-time average performance for operations like `put()` and `get()`.  
  
---  
  
## The Contract Rule  
  
If two objects are considered equal using `equals()`  
(i.e., `a.equals(b) == true`),  
  
then they **must produce the same hashCode**.  
  
Example rule:  
  
if a.equals(b) == true  
then a.hashCode() must be equal to b.hashCode()  
  
Failing to follow this rule can cause incorrect behavior in hash-based collections.  
  
---  
  
## Important Note  
  
The reverse is **not required**.  
  
If:  
  
a.hashCode() == b.hashCode()  
  
it does **not guarantee** that:  
  
a.equals(b) == true  
  
Different objects may produce the same hashCode.   
  
This situation is called a **hash collision** (multiple keys mapping to the same bucket).  
  
---  
  
## What Happens Internally in HashMap  
  
The internal lookup process works as follows:  
  
1. Compute hashCode() of the key  
2. Determine bucket index  
3. Traverse nodes in the bucket  
4. Use equals() to confirm key match  
  
Because of this process:  
  
- `hashCode()` determines the **bucket**  
- `equals()` confirms the **actual key match**  
  
Both methods are required for correct behavior.  
  
---  
  
## Default Behavior from Object  
  
If a class does not override these methods, it inherits the default implementations from `Object`.  
  
Default behavior:  
  
- `equals()` compares memory references  
- `hashCode()` is based on object identity  
  
This means two logically identical objects may still be treated as different.  
  
Example:  
  
    class User {
        String name;
    }

    User u1 = new User();
    u1.name = "Rahul";

    User u2 = new User();
    u2.name = "Rahul";

    System.out.println(u1.equals(u2));

  
Two User objects may contain the same data, but they are different objects in memory.  
Because of this, equals() returns false and they are treated as different elements.  
  
---  
  
## Example Impact in HashSet  
  
If two objects contain identical data but equals() and hashCode() are not overridden:  
  
- HashSet will treat them as different objects  
- Both objects will be stored  
- Duplicate logical data may exist  
  
---  
  
## Best Practice  
  
Whenever equals() is overridden, hashCode() must also be overridden.  
  
This ensures objects behave correctly inside hash-based collections.  
  
---  
  
## Key Interview Insight  
  
Hash-based collections use both methods together:  
  
hashCode() → locate bucket  
equals() → confirm exact key match  
  
Both must be implemented correctly to avoid:  
  
- duplicate entries  
- lookup failures  
- inconsistent behavior
